# 観劇・イベント・飲食ハブ 設計書

## 1. 全体像

3ドメイン（観劇・イベント・飲食店）を単一のデータハブに集約し、静的サイトとして公開、週次でClaudeが推薦を生成する構成です。

```
[データソース] → [Notion（ハブ）] → [GitHub Actions（週次）] → [静的サイト + 週次ダイジェスト]
                                          ↑ Claude API（推薦生成）
```

## 2. ドメイン別の要件とデータ取得可否

| ドメイン | ソース | 取得方法 | 難易度 |
|---|---|---|---|
| 観劇 | 松田観劇カレンダー（公開Googleカレンダー） | iCalフィード or Calendar API。既存ID流用可 | 低 |
| イベント全般 | 現状は散在（Notion、React製トラッカー等） | Notion DBを正とする。登録フローはMCP経由で既存の観劇登録と同型化 | 中 |
| 飲食店 | ローカルガイド投稿 | 注意点あり（下記） | 高 |

飲食店の制約: Googleに「自分の投稿」を取得する公式APIがありません。現実解は2つの併用です。
1. 過去分: Google Takeoutで一括エクスポートし、初回のみNotionへ取り込み
2. 今後分: 現在のレビュー下書きワークフロー（Claudeで210〜230字生成）の完了時に、同じセッションでNotion DBへも保存する運用に変更。これで二重入力なしに蓄積されます

## 3. データモデル（Notion DB 3本）

共通プロパティ: タイトル / 日付（期間）/ エリア / URL / タグ / ステータス（気になる・予定・済）/ 評価 / メモ

- 観劇DB: 劇団・会場・ジャンル（小劇場/ダンス/舞踏 等）を追加
- イベントDB: 種別（展示/フェス/トーク 等）を追加
- 飲食DB: ジャンル・価格帯・再訪意向・レビュー本文を追加

観劇はGoogleカレンダーが正のままでよく、Notionへは同期のみ（カレンダー運用のv4プロトコルは変更不要）。

## 4. サイト設計

- ホスティング: GitHub Pages（無料・Claude Codeとの相性が良い）
- 生成: GitHub ActionsがNotion API + カレンダーフィードからJSONを生成し、静的ビルド
- 画面: ①統合カレンダービュー ②ドメイン別リスト（フィルタ付き）③週次ダイジェストページ ④飲食マップビュー
- 更新頻度: 日次ビルドで十分（イベント情報の鮮度要件は高くない）

## 5. 推薦機構（週次）

GitHub Actionsで毎週金曜などに実行:

1. 入力: 直近登録データ + 未消化の「気になる」+ 嗜好プロファイル
2. 嗜好プロファイルはリポジトリ内のJSONで明示管理（小劇場・コンテンポラリー・舞踏への傾向、フォロー中の劇団/振付家、トマト・玉ねぎ除外、生活圏=代々木上原/渋谷、月間観劇ペース 等）。ブラックボックス化せず自分で編集できるのが利点
3. Claude APIが「今週の推薦5件 + 理由 + 空き日程との突合」を生成
4. 出力先: サイトのダイジェストページ + Gmail送信（MCPまたはAPI）。Instagram Stories用素材への転用も既存パイプラインに接続可能

| フェーズ | 内容 | 目安 |
|---|---|---|
| P1 | 観劇のみでサイト公開（カレンダーフィード→静的サイト） | 1〜2セッション |
| P2 | Notion DB 3本設計 + 飲食のTakeout取り込み + レビュー保存フロー変更 | 2〜3セッション |

## 公開範囲の方針（確定 2026-07-06）

- サイト・リポジトリとも完全公開（GitHub Pages無料プランはpublicリポジトリ必須のため）
- 個人情報はリポジトリに置かない。嗜好プロファイルのうち生活圏・食の除外項目など私的な部分は、GitHub ActionsのSecretsまたはNotion側で管理する
- 観劇の過去アーカイブもサイトに表示する（年別・ジャンル別で閲覧可能に）
- 飲食店データはDB側に「公開/非公開」フラグを持たせ、後から切り替え可能な設計とする
